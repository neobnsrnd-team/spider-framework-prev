## 공통 — 인증·통신 패턴

모든 서버 간 호출은 다음 패턴을 공유한다.

| 요소 | 값 |
|---|---|
| 통신 방식 | REST API (HTTP POST) |
| 콘텐츠 형식 | `application/json` 또는 `multipart/form-data` (파일 전송 시) |
| 인증 헤더 | `x-deploy-token: {DEPLOY_SECRET}` |
| 토큰 비교 | `crypto.timingSafeEqual()` (타이밍 공격 방지) |
| 만료 스케줄러만 | `x-scheduler-token: {SCHEDULER_SECRET}` |

브라우저 호출은 별도로 JWT 세션 쿠키 + Spring Security/`getCurrentUser()` 검증을 거친다.

## 1. 이미지 업로드

**참여자**: 브라우저 → Spider Admin → CMS :3000

**핵심 패턴**: spider-admin은 파일을 디스크에 저장하지 않고 **메모리에서 multipart를 그대로 CMS로 중계**한다.

### 흐름

```
[브라우저]
  multipart/form-data (file + assetName + businessCategory + ...)
       │
       ▼
[Spider Admin: CmsAssetUploadController]
  POST /api/cms-admin/asset-uploads
  - @PreAuthorize('hasAuthority(CMS:W)')
  - @AuthenticationPrincipal에서 userId/userName 추출 (클라이언트 입력 무시)
  - assetUploadValidator.validate(file)
       │
       ▼
[Spider Admin: CmsBuilderClient.upload()]
  - MultipartFile.getResource()로 스트림 참조만 사용 (메모리 효율)
  - UTF-8 명시 (한글 파일명 깨짐 방지)
  - x-deploy-token 헤더 주입
  - RestClient POST → CMS
       │
       ▼
[CMS: /cms/api/builder/upload]
  - SERVER_MODE === 'operation' 이면 차단
  - x-deploy-token 검증 → 서버 간 호출로 식별
  - file.arrayBuffer() → Buffer 변환
  - assetId = crypto.randomUUID()
  - filename = `{assetId}_{safeFilename}`
  - fs.writeFile(ASSET_UPLOAD_DIR/filename)
  - createAsset() → SPW_CMS_ASSET INSERT (ASSET_STATE = 'WORK')
  - 실패 시 fs.unlink() 보상
       │
       ▼
[응답]
  { ok: true, data: { url, assetId } } → spider-admin → 브라우저
```

### 관련 파일

| 위치 | 파일 |
|---|---|
| spider-admin 컨트롤러 | `domain/cmsasset/controller/CmsAssetUploadController.java` |
| spider-admin 서비스 | `domain/cmsasset/service/CmsAssetService.java` |
| spider-admin 클라이언트 | `domain/cmsasset/client/CmsBuilderClient.java` |
| CMS API | `src/app/api/builder/upload/route.ts` |
| CMS DB 레이어 | `src/db/repository/asset.repository.ts` |

## 2. 이미지 승인 (Saga 패턴)

**참여자**: Spider Admin → CMS :3000 (파일 이동)

**핵심 패턴**: DB 상태 변경과 CMS 파일 이동을 **두 단계 분리 트랜잭션 + 보상 롤백**으로 처리한다. 단일 `@Transactional`로 묶으면 CMS가 커밋 전 PENDING만 보게 되어 항상 거절된다.

### 흐름

```
[Spider Admin: CmsAssetService.approve()]
       │
       ▼
   ① 독립 TX 커밋
      transactionTemplate.executeWithoutResult(...)
        - assertTransition(PENDING → APPROVED)
        - cmsAssetMapper.updateState(assetId, PENDING, APPROVED)
        - 실패 시 InvalidStateException → 롤백
      [TX 커밋 완료] ← 여기서 DB는 APPROVED 상태
       │
       ▼
   ② CMS 파일 배포 호출
      cmsBuilderClient.deployAsset(assetId)
        POST /cms/api/assets/{assetId}/deploy (read-timeout 10초)
       │
       ▼
[CMS: /cms/api/assets/{assetId}/deploy]
   - x-deploy-token 검증
   - getAssetById(assetId)
   - ASSET_STATE === 'APPROVED' 확인 ← ①이 커밋되어 있어야 통과
   - 멱등성 — ASSET_PATH가 이미 deployedPath면 조기 반환
   - mkdir(/data/deployed/img/, recursive)
   - fs.copyFile(/data/uploads/{file} → /data/deployed/img/{file})
   - updateAssetPathUrl() — DB의 ASSET_PATH·ASSET_URL을 deployed 경로로 갱신
   - fs.unlink(/data/uploads/{file}) — 원본 삭제 (실패 시 경고만)
   - { ok: true, data: { url } }
       │
       ▼
[Spider Admin: 정상 종료] log.info('승인 및 배포 완료')

────────────────────────────────────────────────────────────

[CMS 호출 실패 시 — Saga 보상]
   ③ catch (BaseException deployEx)
      transactionTemplate.executeWithoutResult(...)
        - cmsAssetMapper.updateState(assetId, APPROVED, PENDING)
      throw deployEx
      → 보상 실패 시 데이터-파일 불일치, error 로깅 (수동 복구)
```

### 왜 Saga인가

```
잘못된 설계: 단일 TX로 묶기
  @Transactional {
    DB UPDATE PENDING → APPROVED   ← 아직 커밋 안 됨
    HTTP POST /cms/.../deploy       ← CMS는 여전히 PENDING을 읽음 → 400 거절
  }

올바른 설계: Saga 패턴
  TX1: DB UPDATE 커밋 (CMS가 APPROVED를 읽을 수 있게)
  HTTP POST → CMS 처리
  실패 시 TX2: DB 보상 롤백 (APPROVED → PENDING)
```

### 관련 파일

| 위치 | 파일 |
|---|---|
| spider-admin 서비스 | `domain/cmsasset/service/CmsAssetService.java` (approve 메서드) |
| CMS API | `src/app/api/assets/[assetId]/deploy/route.ts` |

## 4. 페이지 승인

**참여자**: Spider Admin → CMS :3000 (DB UPDATE만, 파일 이동 없음)

이미지와 달리 페이지는 **승인 시점에 파일 이동이 없다.** HTML은 DB CLOB에만 있고, 별도 "배포" 액션을 통해서만 운영 서버에 파일로 떨어진다.

### 흐름

```
[Spider Admin: 결재자 승인 클릭]
       │
       ▼
[POST /cms/api/builder/pages/{pageId}/approve]
  - x-deploy-token 검증
  - 페이지 조회 → APPROVE_STATE === 'PENDING' 확인
  - UPDATE SPW_CMS_PAGE SET APPROVE_STATE = 'APPROVED'
       │
       ▼
[응답] { ok: true }
```

승인 후 별도 [페이지 배포](#5-페이지-배포-pushreceive) 단계가 필요하다.

## 5. 페이지 배포 (push/receive)

**참여자**: CMS :3000 (push) → CMS :3001 (receive, 다른 운영 서버들에도 동시 전송)

**핵심 패턴**: HTML 프래그먼트를 완전한 HTML 문서로 조립한 뒤, **활성 운영 서버 전체에 병렬 전송**한다.

### 흐름

```
[POST /cms/api/deploy/push]
  - 인증: x-deploy-token (서버 간) 또는 세션 권한 (브라우저)
  - 페이지 조회 → APPROVE_STATE === 'APPROVED' 확인
       │
       ▼
   ① HTML 조립 — buildDeployHtml(fragment, pageId, pageName)
      - public/runtime/contentbuilder-runtime.css 읽어서 인라인
      - LAYOUT_CSS 인라인
      - 에셋 경로 치환:
          src="/assets/..." → src="{CMS_BASE_URL}/cms/assets/..."
          src="/uploads/..." → src="{CMS_BASE_URL}/cms/uploads/..."
          url(/assets/...) → url({CMS_BASE_URL}/cms/assets/...)
      - 완전한 <!DOCTYPE html>...</html> 문서로 감싸기
      - ContentBuilder 런타임 스크립트 + 플러그인 동적 로드 코드 추가
      - cms-tracker.js 스크립트 태그 추가 (페이지뷰 추적용)
       │
       ▼
   ② CRC 계산 — SHA-256 앞 16자리
       │
       ▼
   ③ ALIVE_YN='Y' 서버 목록 조회 — getServerList('Y')
       │
       ▼
   ④ 버전 결정
      - FILE_PATH 있으면 getHistoryVersionByFilePath()
      - 없으면 getLatestHistory().VERSION
      - fileId = `{pageId}_v{version}.html`
       │
       ▼
   ⑤ 각 서버에 병렬 전송 (Promise.all)
      forEach(server in servers):
        sendToServer(server.INSTANCE_IP:PORT/cms/api/deploy/receive, pageId, html, trackerJs)
        upsertFileSend({ instanceId, fileId, fileSize, fileCrcValue, ... })
                ▼
        ┌───────────────────────────────────────┐
        │ [CMS :3001 — POST /cms/api/deploy/receive]
        │   - x-deploy-token 검증
        │   - 경로 트래버설 방지 (pageId에 .. / \ 차단)
        │   - mkdir(public/deployed/, recursive)
        │   - fs.writeFile(public/deployed/{pageId}.html, html)
        │   - trackerJs 있으면 public/cms-tracker.js 갱신
        │   - { ok: true, data: { path } }
        └───────────────────────────────────────┘
       │
       ▼
   ⑥ 하나 이상 성공하면 updatePageDeploy() — 페이지 마지막 배포 정보 갱신
       │
       ▼
[응답]
  { ok: true, data: { fileId, crcValue, successCount, failCount, results } }
```

### 배포 이력 — Spider Admin과의 연동

`upsertFileSend()`는 `FWK_CMS_FILE_SEND_HIS` 테이블에 INSERT/UPDATE한다. 이 테이블은 **spider-admin 스키마**에 있어, CMS가 직접 spider-admin DB에 배포 이력을 기록하는 구조다.

### 관련 파일

| 위치 | 파일 |
|---|---|
| Push API | `src/app/api/deploy/push/route.ts` |
| Receive API | `src/app/api/deploy/receive/route.ts` |
| 전송 유틸 | `src/lib/deploy-utils.ts` (`sendToServer`, `buildServerUrl`) |
| 이력 저장 | `src/db/repository/file-send.repository.ts` |

## 6. A/B 테스트 라우팅

**참여자**: 사용자 브라우저 → CMS :3001 (`/cms/api/ab/{groupId}`)

**핵심 패턴**: **가중치 기반 무작위 선택 + 쿠키 기반 sticky session.**

### 흐름

```
[GET /cms/api/ab/{groupId}]
       │
       ▼
   ① getAbGroup(groupId) — 그룹 내 페이지 목록 조회
      필터: IS_PUBLIC = 'Y' AND AB_WEIGHT > 0
      비어있으면 404
       │
       ▼
   ② 쿠키 검증 — `ab_{groupId}` 쿠키 확인
      - 쿠키 있고 그룹 내 유효한 페이지면 → savedPageId 사용 (sticky)
      - 없거나 무효 → pickByWeight()
       │
       ▼
   ③ pickByWeight(pages)
      - candidates = AB_WEIGHT > 0 인 페이지
      - total = sum(AB_WEIGHT)
      - rand = Math.random() * total
      - 누적 가중치로 선택, 부동소수점 오차 시 마지막 후보로 폴백
       │
       ▼
   ④ 운영 서버 결정
      - getServerList('Y')
      - localhost / 127.0.0.1 / ::1 제외 (시드 데이터의 React 데모 서버 회피)
      - INSTANCE_IP가 비어있는 서버도 제외 (잘못된 'http:///' URL 방지)
      - 폴백: 그래도 없으면 503
       │
       ▼
   ⑤ 리다이렉트
      302 → http://{operationServer.IP}:{PORT}/cms/deployed/{targetPageId}.html
      신규 배정 시 res.cookies.set('ab_{groupId}', targetPageId, { httpOnly, sameSite: 'lax', maxAge: 30일 })
```

### Sticky Session의 의미

- **일반 사용자**: 처음 접속 시 가중치에 따라 A/B 배정 → 30일간 동일 페이지 (통계적 유효성 확보)
- **시크릿 모드**: 쿠키 없는 신규 세션 → 매번 새로 추첨 (관리자 검수에 유용)

### 관련 파일

| 위치 | 파일 |
|---|---|
| A/B 라우팅 API | `src/app/api/ab/[groupId]/route.ts` |
| 그룹 조회 | `src/db/repository/page.repository.ts#getAbGroup` |
| 서버 목록 | `src/db/repository/file-send.repository.ts#getServerList` |

## 7. 페이지뷰·클릭 추적

**참여자**: 배포된 페이지 → CMS :3000 (트래커 API)

배포된 HTML에 `cms-tracker.js`가 포함되어 있어, 페이지가 로드되면 자동으로 트래커 API를 호출한다.

### 흐름

```
[배포된 페이지 로드 (운영 서버 :3001)]
  <script src="{CMS_BASE_URL}/cms/cms-tracker.js" data-page-id="..." data-cms-url="...">
       │
       ▼
   ① 페이지뷰 — POST /cms/api/track/view
      body: { pageId, sessionId, referrer, userAgent, ... }
      → SPW_CMS_PAGE_VIEW_LOG INSERT
       │
       ▼
   ② 클릭 (선택) — 사용자 클릭 시 POST /cms/api/track/click
      body: { pageId, elementSelector, sessionId, ... }
      → SPW_CMS_CLICK_LOG INSERT

[관리자 통계 조회]
   GET /cms/api/track/stats?pageId=...&from=...&to=...
   → 집계 쿼리로 페이지뷰·클릭 수 반환
```

### 크로스오리진 처리

운영 서버(:3001/8080)에서 트래커가 관리 서버(:3000/80)의 트래커 API를 호출하므로 **크로스오리진**이다. nginx에서 `/cms/api/track/` 경로에 대해 CORS 헤더를 응답한다.

```nginx
location ~ ^/cms/api/track/ {
    if ($request_method = 'OPTIONS') { ... return 204; }
    proxy_hide_header 'Access-Control-Allow-Origin';
    add_header 'Access-Control-Allow-Origin' '*' always;
    proxy_pass http://localhost:3000;
}
```

### 관련 파일

| 위치 | 파일 |
|---|---|
| 트래커 스크립트 | `public/cms-tracker.js` |
| 페이지뷰 API | `src/app/api/track/view/route.ts` |
| 클릭 API | `src/app/api/track/click/route.ts` |
| 통계 API | `src/app/api/track/stats/route.ts` |
| nginx CORS | `nginx/nginx.conf` |

## 8. 만료 스케줄러

**참여자**: 내장 cron(node-cron) → CMS 자체 → 운영 서버들

**핵심 패턴**: 만료된 페이지를 발견하면 **만료 안내 HTML로 일괄 교체 배포**한다.

### 흐름

```
[node-cron — 주기적 실행]
       │
       ▼
[POST /cms/api/scheduler/expire (또는 직접 runExpireJob() 호출)]
  - x-scheduler-token 검증 (외부 호출 시)
       │
       ▼
   ① runExpireJob() — src/lib/scheduler.ts
      - 만료 대상 페이지 조회 (EXPIRE_DT < NOW AND IS_PUBLIC = 'Y')
       │
       ▼
   ② 각 만료 페이지마다 deployExpiredPage(pageId)
      - public/system/page-expired.html 읽기
      - getServerList('Y')
      - fileId = `{pageId}_v{version}_expired.html` (정상 배포 이력 덮어쓰기 방지)
      - forEach(server):
          sendToServer(server.../deploy/receive, pageId, expiredHtml)
          upsertFileSend({ ..., lastModifierId: 'scheduler' })
       │
       ▼
   ③ 페이지 IS_PUBLIC = 'N' 으로 비활성화
       │
       ▼
[응답] { processed, failed }
```

### 왜 별도 fileId 접미사를 쓰나

`{pageId}_v{version}_expired.html` 처럼 `_expired` 접미사를 붙이는 이유는 정상 배포 이력(`{pageId}_v{version}.html`)을 덮어쓰지 않기 위함이다. 만료 처리 후 다시 활성화할 때 원래 버전 이력이 보존된다.

### 관련 파일

| 위치 | 파일 |
|---|---|
| 스케줄러 진입점 | `src/instrumentation.ts` (Next.js 서버 시작 시 cron 등록) |
| 비즈니스 로직 | `src/lib/scheduler.ts` |
| 외부 호출 API | `src/app/api/scheduler/expire/route.ts` |
| 만료 안내 페이지 | `public/system/page-expired.html` |

## 부록 — 공통 트랜잭션·DB 패턴

### CMS (Next.js + oracledb)

```typescript
import { withTransaction } from '@/db/connection';

await withTransaction(async (conn) => {
    await conn.execute(SQL_UPDATE_A, { ... });
    await conn.execute(SQL_INSERT_B, { ... });
    // 예외 발생 시 자동 ROLLBACK, 정상 종료 시 COMMIT
```

### 토큰 검증 (양 프로젝트 공통)

```typescript
// CMS
import { timingSafeEqual } from 'crypto';
function isValidToken(token) {
    if (!DEPLOY_SECRET || !token) return false;
    const expected = Buffer.from(DEPLOY_SECRET, 'utf8');
    const received = Buffer.from(token, 'utf8');
    if (expected.length !== received.length) return false;
    return timingSafeEqual(expected, received);
}
```
