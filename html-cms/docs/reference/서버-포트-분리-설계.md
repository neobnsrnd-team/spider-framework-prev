# CMS 서버 포트 분리 설계 — :3000 / :3001

> 작성일: 2026-05-06

---

## 한 줄 요약

**포트 분리의 목적은 세 가지다: (1) 미승인 이미지의 배포 환경 유입을 OS 레벨에서 원천 차단, (2) 관리자 서버를 인터넷에서 물리적으로 분리, (3) 대고객 트래픽과 관리자 작업의 독립적 스케일.**

이 세 가지는 코드 레벨 권한 체크만으로는 대체할 수 없는 인프라 수준의 보장이다.

---

## Docker 이미지 하나로 서버가 두 개인 원리

### 이미지 vs 컨테이너

```
이미지    = 레시피 (설계도, 실행 파일 묶음)
컨테이너  = 레시피로 만든 실제 실행 중인 프로세스
```

같은 레시피로 음식을 여러 개 만들 수 있듯이, **하나의 이미지로 컨테이너를 몇 개든 실행할 수 있다.** 각 컨테이너는 독립된 프로세스로, 자기만의 메모리·파일시스템·네트워크를 가진다.

### 지금 서버에서 실제로 돌아가는 것

```
[호스트 서버 (NHN Cloud 인스턴스)]
│
├── 컨테이너 A: springware-cms        ← 이미지: springware-cms:latest
│     PORT=3000, SERVER_MODE=cms
│     /data/uploads → 읽기+쓰기
│
└── 컨테이너 B: springware-operation  ← 이미지: springware-cms:latest (동일!)
      PORT=3000, SERVER_MODE=operation
      /data/uploads → 읽기 전용
```

같은 이미지지만 두 컨테이너는 완전히 별개의 프로세스로 실행된다.

### 포트 매핑 — 외부에서 다른 번호로 접근하는 이유

컨테이너 내부에서는 둘 다 3000번 포트로 실행된다. 그런데 호스트 서버 바깥에서 보면 포트 번호가 달라진다. 이것이 Docker **포트 매핑**이다.

```yaml
# docker-compose-prod.yml
cms:
  ports:
    - "3000:3000"   # 호스트 3000 → 컨테이너 내부 3000

operation:
  ports:
    - "3001:3000"   # 호스트 3001 → 컨테이너 내부 3000
```

컨테이너 B는 내부에서 3000으로 듣고 있지만, 외부에서는 3001로 접근해야 한다.

### 같은 코드인데 동작이 다른 이유 — 환경변수

컨테이너 실행 시 **환경변수**를 다르게 주입한다.

```yaml
cms:
  environment:
    SERVER_MODE: "cms"        # 이미지 업로드 허용

operation:
  environment:
    SERVER_MODE: "operation"  # 이미지 업로드 차단
```

코드 안에서 이 값을 읽어서 분기한다:

```typescript
if (process.env.SERVER_MODE === 'operation') {
    // 업로드 거부
}
```

같은 코드지만 환경변수에 따라 다르게 동작하는 구조다. 앱 설치 파일은 하나인데 설정 파일만 다르게 주는 것과 같다.

### nginx가 트래픽을 나누는 방법

컨테이너가 두 개이므로 nginx가 어느 컨테이너로 보낼지 결정한다.

```
사용자 요청 → nginx (문지기)
  :80   → 컨테이너 A (cms, 관리자용)
  :8080 → 컨테이너 B (operation, 대고객용)
```

### 전체 구조 한눈에 보기

```
                   [호스트 서버]
                        │
            ┌───── nginx (문지기) ─────┐
            │                         │
          :80                       :8080
            │                         │
    [컨테이너 A]                [컨테이너 B]
    springware-cms          springware-operation
    SERVER_MODE=cms         SERVER_MODE=operation
    uploads: 읽기+쓰기      uploads: 읽기 전용
       (같은 이미지)            (같은 이미지)
```

## 왜 포트를 분리했는가

CMS는 동일한 소스코드·동일한 Docker 이미지로 두 컨테이너를 띄운다. 포트가 다른 이유는 **기능이 달라서가 아니라, 파일시스템 접근 권한을 다르게 줘야 하기 때문**이다.

### 두 서버의 역할

| 구분 | :3000 (cms 모드) | :3001 (operation 모드) |
|---|---|---|
| 대상 사용자 | 현업 관리자 (콘텐츠 제작자) | 현업 제작자 + 일반 사용자 |
| 주요 업무 | 이미지 업로드, 빈 화면 템플릿 제작 | 템플릿 기반 페이지 제작, 대고객 콘텐츠 서빙 |
| 이미지 업로드 | **가능** | **불가** (OS 레벨 차단) |
| `/data/uploads/` 마운트 | 읽기+쓰기 | **읽기 전용 (:ro)** |
| `/data/deployed/` 마운트 | 없음 | 읽기+쓰기 |

### 핵심: OS 레벨 파일시스템 격리

```yaml
# docker-compose-prod.yml

cms:        # :3000
  volumes:
    - /data/uploads:/app/public/uploads       # 쓰기 가능
    - /data/deployed:/app/public/deployed

operation:  # :3001
  volumes:
    - /data/uploaded:/app/public/uploads:ro   # 읽기 전용 — 쓰기 시도 시 OS가 거부
    - /data/deployed:/app/public/deployed
```

`:ro` 마운트는 애플리케이션 코드나 권한 체크와 무관하게, **운영체제가 직접 쓰기를 거부**한다. 버그, 코드 우회, 잘못된 권한 설정이 있어도 미승인 이미지가 배포 환경에 올라가는 것이 원천 차단된다.

## 저장소 구조와 흐름

```
/data/uploads/          ← 미승인 이미지 보관소 (:3000 쓰기, :3001 읽기)
/data/deployed/
    img/                ← 승인된 이미지 (:3001 쓰기)
    {pageId}.html       ← 배포된 페이지 HTML (:3001 쓰기)
```

### 승인 전/후 이미지 이동 흐름

```
현업 관리자 (:3000)
  이미지 업로드 → /data/uploads/ 저장 (미승인)
                        ↓
              Spider Admin에서 결재자 승인
                        ↓
        /data/uploads/ → /data/deployed/img/ 복사 (승인 완료)
                        ↓
        :3001 에디터에서 승인된 이미지만 사용 가능
```

미승인 이미지는 절대로 `/data/deployed/`에 들어갈 수 없다. 승인 API가 명시적으로 복사하기 전까지는 `/data/uploads/`에만 존재하고, :3001은 거기에 쓰기를 할 수 없다.

## 코드 레벨 분기 (SERVER_MODE)

포트 분리 외에도 `SERVER_MODE` 환경변수로 동작을 분기한다.

```
SERVER_MODE=cms        → :3000: 이미지 업로드 허용, 빈 화면 시작 허용
SERVER_MODE=operation  → :3001: 이미지 업로드 API 차단, 빈 화면 시작 제한(예정)
```

OS 레벨 차단(`:ro` 마운트)이 1차 방어선이고, `SERVER_MODE` 코드 분기는 2차 방어선이다. 두 겹으로 미승인 리소스 유입을 막는 구조다.

## "같은 코드인데 포트만 다른 거 아닌가요?"에 대한 답

맞다. 소스코드와 Docker 이미지는 동일하다. 포트 자체는 수단일 뿐이고, **실질적인 차이는 Docker 볼륨 마운트 설정과 네트워크 접근 제어에 있다.**

포트를 달리 쓰는 이유는 nginx 리버스 프록시에서 두 컨테이너를 외부에 노출할 때 포트로 구분해야 방화벽 레벨에서 접근 제어가 가능하기 때문이다.

```
인터넷   → nginx :8080 → :3001 (대고객 서빙)    ← 외부 공개
사내망   → nginx :80   → :3000 (관리자 전용)    ← 방화벽으로 사내에서만 접근
```

단일 포트를 쓰면 관리자 기능이 있는 서버가 인터넷에 노출될 수밖에 없다. 포트 분리가 되어야 방화벽 규칙 하나로 관리자 서버 자체를 사내망에만 가둘 수 있다.

파일시스템 측면에서도, 컨테이너를 하나로 합치면 코드 버그 하나로 미승인 이미지가 배포 환경에 올라갈 수 있다. `:ro` 마운트로 이를 인프라 구조에서 막는 것이 분리의 또 다른 목적이다.

## 현재 구현 상태 (2026-05-06 기준)

### 구현 완료

| 항목 | 상태 |
|---|---|
| Docker 볼륨 격리 (`:ro` 마운트) | ✅ 완료 |
| SERVER_MODE 이미지 업로드 차단 | ✅ 완료 |
| nginx 포트별 라우팅 | ✅ 완료 |

### 미구현 — 분리의 실효성을 높이는 항목

| 항목 | 설명 |
|---|---|
| 역할 세분화 | 현재 2단계(admin/user) → 현업 관리자(:3000 전용) / 현업 제작자(:3001 전용) 분리 |
| :3001 빈 화면 시작 차단 | 코드 레벨에서 SERVER_MODE=operation 시 빈 화면 생성 불가 처리 |
| :3000 접근 경로 정비 | Spider Admin에서 관리자용 에디터 링크를 :3000으로 분기 |

현재는 Spider Admin의 에디터 링크가 전부 :3001(8080 포트)을 향하고 있어, 설계상 :3000에서만 해야 할 템플릿 제작도 :3001에서 이루어지고 있다. 볼륨 격리는 이미 적용되어 있으므로 이미지 업로드 차단은 정상 동작하지만, 역할 기반 접근 제어는 미구현 상태다.

역할 세분화 구현 시점에 아래를 함께 정비한다.

```
Spider Admin 로그인
    ↓
역할 확인
    ↓
현업 관리자 → :3000 (80포트) 에디터 링크 제공
              이미지 업로드 가능, 빈 화면 템플릿 제작 가능

              승인된 템플릿 선택 → 페이지 제작
              이미지 업로드 불가, 빈 화면 시작 불가
```

## 역할과 기능이 모두 구현되어도 포트 분리가 맞는가

**그렇다. 모든 기능이 구현된 이후에도 포트 분리 유지가 올바른 방향이다.**

코드 레벨 권한 체크는 "이 API를 호출할 수 있는가"를 제어하지만, 포트 분리는 그보다 낮은 레벨에서 세 가지를 동시에 보장한다.

### 1. 네트워크 레벨 격리 — 코드로 대체 불가

```
인터넷 차단 ───────────────────────────────────────────┐
                                                      │ 방화벽
사내망 ──→ nginx :80 ──→ :3000 (관리자 서버)          │ 규칙
인터넷 ──→ nginx :8080 → :3001 (대고객 서버) ─────────┘
```

단일 서버를 쓰면 관리자 기능이 인터넷에 노출된 포트에 존재하게 된다. 방화벽으로 특정 엔드포인트만 차단하는 것은 URL 패턴 기반이라 누락 위험이 있다. 서버 자체를 사내망에만 두는 것이 훨씬 강한 보장이다.

### 2. 트래픽 독립 스케일

대고객 서버(:3001)는 일반 사용자 트래픽이 집중된다. 단일 서버라면 고객 트래픽 급증 시 관리자 작업도 함께 느려진다. 포트 분리 상태에서는 :3001만 수평 확장하면 된다.

### 3. OS 레벨 파일 격리 — 코드 버그 방어

코드 권한 체크는 버그가 생길 수 있다. `:ro` 마운트는 코드와 무관하게 OS가 쓰기를 거부한다.

---

### 단일 서버가 충분한 경우

사내 내부 도구이고 네트워크 분리 계획이 없다면 단일 서버 + 코드 레벨 권한 체크도 실용적으로 충분하다. 운영 복잡도를 낮추는 장점이 있다.

**판단 기준**: 관리자 서버를 인터넷에서 물리적으로 격리할 필요가 있는가. 금융권처럼 보안 요건이 높다면 포트 분리 유지가 맞고, POC·소규모 사내 도구 수준이라면 단일 서버로 단순화해도 된다.
