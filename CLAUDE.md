# CLAUDE.md

## GitHub 작업 규칙

### 이모지 & 커밋 유형 시스템

| 이모지 | 유형        | 설명                         |
|--------|-------------|------------------------------|
| ✨     | FEATURE     | 신규 기능 개발               |
| 🐛     | FIX         | 버그 수정                    |
| ⚡️    | PERFORMANCE | 성능 개선                    |
| ♻️     | REFACTOR    | 코드 리팩토링                |
| 📝     | DOCS        | 문서 작업 (README 등)        |
| 🎨     | STYLE       | UI 디자인 및 스타일 수정     |
| 🧪     | TEST        | 테스트 코드 추가 및 수정     |
| 🔐     | SECURITY    | 보안 관련 패치               |
| 🚀     | DEPLOY      | 배포 및 인프라 설정          |

커밋 메시지 형식: `<이모지> <TYPE>: <한 줄 설명>`
예) `🎨 STYLE: Tailwind 테마 변수 통합 및 임의 색상 제거`

### 이슈 생성

- 이슈 생성 시 반드시 `.github/ISSUE_TEMPLATE/feature_request.md` 템플릿을 기반으로 작성한다.
- 이슈 생성 시 label은 `.github/label.json`을 사용한다.
- 제목 형식: `<이모지> <TYPE>: <한글 핵심 설명>`

### 브랜치 생성

형식: `<type>/#<issue-number>`

- type은 소문자: `feat` / `fix` / `perf` / `refactor` / `docs` / `style` / `test` / `security` / `deploy`
- 예) `feat/#12`, `style/#34`

### PR 생성

- PR 생성 시 반드시 `.github/pull_request_template.md` 템플릿을 기반으로 작성한다.
- PR 생성 시 label은 `.github/label.json`을 사용한다.
- PR 제목 형식: `<이모지> <type>/#<issue-number>-<한글 핵심키워드>`

#### 선택 섹션 작성 기준

템플릿의 `(선택)` 섹션은 아래 조건에 따라 자율적으로 채운다:
- `🛠 기술적 의사결정` — 여러 구현 방식 중 하나를 선택한 이유가 있을 때
- `🧪 테스트` — 검증 방법이나 테스트 코드를 설명할 필요가 있을 때
- `📸 변경 사항 확인` — UI 관련 파일(*.tsx, *.css, *.vue) 변경이 감지될 때
- `⚠️ 고려 및 주의 사항` — 사이드 이펙트, 마이그레이션, 배포 유의점이 있을 때
- `💬 리뷰 포인트` — 특별히 의견이 필요한 부분이 있을 때
- `🔗 참고 사항` — 관련 문서, 링크, 추가 전달 사항이 있을 때

### Claude 행동 원칙

1. 사용자가 작업 의도를 말하면 이슈 생성 → 브랜치 생성 순서대로 실행 제안
2. `gh` CLI 명령 실행 전 반드시 명령 내용을 사용자에게 보여주고 확인 후 실행
3. 모든 작업 완료 후 판단 보고 한 줄 제시
4. 이슈 번호가 없는 경우 이슈 먼저 생성 후 브랜치/PR 진행
5. `git status`, `git diff` 결과를 직접 분석하여 PR 본문 자동 생성

## 코드 작성 규칙

### 주석

코드만 봐도 의도를 파악할 수 있도록 주석을 충분히 작성한다.

**파일 상단 JSDoc**

- 새 파일을 생성할 때는 반드시 파일 상단에 JSDoc 주석을 작성한다.
- 주석에는 다음 내용을 포함한다:
  - 파일명 (`@file`)
  - 파일/함수의 역할 설명 (`@description`)
  - 주요 파라미터 (`@param`)
  - 반환값 (`@returns`)
  - 필요한 경우 사용 예시 (`@example`)

** 파일 상단 JavaDoc**:
  - 클래스: 역할 한 줄 요약 + 필요 시 `<p>` 태그로 상세 설명 (설정 클래스는 사용 예시 `<pre>{@code ...}</pre>` 포함)
  - 필드: `/** 한 줄 설명 */` 형식. 단순 getter/setter는 생략
  - 내부 로직이 자명하지 않은 private 메서드에도 작성
  - 비즈니스 로직 설명이 없는 단순 위임 메서드(Controller → Service 단순 호출 등)는 생략 가능

**인라인 주석**

- 다음 경우에는 반드시 인라인 주석을 추가한다:
  - 왜 이 값을 선택했는지 이유가 필요한 상수·기본값 
  - 개발자가 처음 봤을 때 의도를 오해할 수 있는 로직
  - 타입 단언(`as`)이나 예외 처리 등 방어 코드
  - 여러 분기 중 특정 분기가 존재하는 이유가 명확하지 않은 경우

# DB 테이블 신규 / 삭제 / 변경 규칙
- DB 이력 관리를 위해 프로젝트 파일에 쿼리를 작성한다.
- 쿼리 수행은 개발자가 DB에서 직접 수행해야한다는 점을 명시한다.
- /batch-was 프로젝트 제외한 다른 프로젝트 작업 중 테이블에 신규 / 삭제 / 변경이 필요한 경우 admin/docs/sql/oracle/ 하위 .sql 파일에 쿼리를 작성한다.  
- /batch-was 프로젝트 작업 중 테이블 신규 / 삭제 / 변경 사항의 경우 batch-was/docs/sql 하위 .sql 파일에 쿼리를 작성한다. .sql 신규 파일을 생성하지 않고 기존에 있는 파일에 쿼리를 추가한다.
